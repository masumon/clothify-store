[MEDIUM-01] Admin mutation routes do not perform explicit CSRF / Origin validation
------------------------------------------------------------
Evidence:
- proxy.ts:28-45
- lib/admin-auth.ts:160-165
- app/api/admin/products/route.ts:29-82
- app/api/admin/products/route.ts:89-160
- app/api/admin/settings/route.ts:19-51
- app/api/admin/upload/route.ts:43-91
- app/api/admin/orders/[id]/status/route.ts:14-38
- Repo-wide search for "origin|referer|csrf" across app/api/admin and app/api/orders returned no explicit handler-level checks

Why this matters:
- Admin mutations are protected mainly by session cookie + proxy auth gate
- Session cookie uses SameSite=Lax, which reduces common cross-site abuse
- কিন্তু explicit Origin / Referer / anti-CSRF token verification না থাকায় protection entirely browser cookie policy-র উপর নির্ভর করছে
- Defense-in-depth হিসেবে authenticated admin POST/PATCH/PUT/DELETE route-এ explicit origin validation থাকা উচিত

Practical risk:
- Real-world exposure is lower than a fully unprotected cookie-auth app because sameSite=lax is present
- তবুও এটি still an unresolved hardening gap, especially for future browser edge-cases, browser extensions, internal tooling, or same-site subdomain scenarios

Recommended remediation:
1. Shared helper বানিয়ে state-changing admin route-এ trusted origin check enforce করুন
2. Optionally `x-clothify-admin-csrf` custom header + double-submit token add করুন
3. Proxy/auth layer-এ mutation method (`POST`, `PUT`, `PATCH`, `DELETE`) special handling যোগ করুন

Status:
- Open

[MEDIUM-02] Admin authentication still supports shared static credentials and HTTP Basic fallback
------------------------------------------------------------
Evidence:
- proxy.ts:19-20
- proxy.ts:33-35
- lib/admin-auth.ts:44-56
- lib/admin-auth.ts:179-201
- lib/admin-auth.ts:213-218
- app/api/admin/auth/route.ts:92-127
- lib/api/admin-health.ts:132-146

Why this matters:
- Entire admin surface still depends on one shared env username/password pair
- HTTP Basic fallback remains valid for admin paths/APIs
- Browser/client/tooling may cache Authorization header
- If `ADMIN_SESSION_SECRET` is missing, session secret can fall back to `username:password` derivation (`lib/admin-auth.ts:51-56`)

Current mitigation already present:
- constant-time compare
- lockout on password auth
- signed session cookie
- optional Supabase bridge path exists

Remaining concern:
- Shared admin identity weakens accountability
- Static credential model makes rotation/manual sharing operationally risky
- Basic auth is useful for backward compatibility, but security posture would be stronger if it were disabled by default in production

Recommended remediation:
1. Disable HTTP Basic fallback in production unless a concrete machine-client use-case requires it
2. Make `ADMIN_SESSION_SECRET` mandatory at startup instead of optional fallback
3. Prefer per-user Supabase admin identities and authorized email allowlist for long-term admin access
4. Add admin action audit trail keyed to user identity, not a single shared env user

Status:
- Open, partially mitigated

[MEDIUM-03] No explicit CSP / browser security headers are defined in repo-level app config
------------------------------------------------------------
Evidence:
- next.config.js:23-45
- vercel.json:1-3
- app/layout.tsx:93-101

Why this matters:
- Root layout loads external CDN stylesheet and an inline bootstrap script
- Repo-level Next.js config does not define `headers()`
- Vercel config does not define browser security headers either
- This weakens defense-in-depth for XSS, asset injection, MIME confusion, clickjacking, and policy governance

Specific posture observations:
- No visible CSP in app config
- No visible `X-Content-Type-Options: nosniff` rule in repo config
- No visible `Referrer-Policy` rule in repo config
- No visible `frame-ancestors` / clickjacking policy in repo config
- Inline script in root layout will make future strict CSP harder unless moved to nonce/hash-based model

Recommended remediation:
1. Add security headers in Next config or edge config
2. Prefer a CSP strategy that supports:
   - self assets by default
   - explicit font/image/script allowlist
   - frame-ancestors restriction
3. Move inline bootstrap logic to a nonce-backed or hashed script strategy
4. Self-host Font Awesome or replace with local SVG/icon pipeline

Status:
- Open

[LOW-01] Invoice popup writes raw HTML assembled from dynamic order/store fields
------------------------------------------------------------
Evidence:
- components/InvoiceAutoDownload.tsx:90-110
- components/InvoiceAutoDownload.tsx:112-256
- components/InvoiceAutoDownload.tsx:203-214
- components/InvoiceAutoDownload.tsx:305-309
- components/InvoiceAutoDownload.tsx:343-347

Why this matters:
- `customerName`, `address`, `item.name`, `storeName`, `storeAddress`, `storePhone` ইত্যাদি HTML string interpolation-এর মাধ্যমে সরাসরি printable markup-এ ঢুকছে
- এরপর `popup.document.write(html)` দিয়ে render করা হচ্ছে
- এটি avoidable self-XSS / DOM injection pattern

Practical impact:
- Current exposure likely limited mostly to the customer’s own browser session in the order-success / invoice flow
- তবে unsafe string-to-HTML pattern future reuse হলে stored rendering issue-তেও রূপ নিতে পারে

Recommended remediation:
1. Interpolated text HTML-escape করুন
2. Prefer a dedicated printable route/component instead of raw `document.write`
3. PDF/invoice generation path-এ template escaping centralize করুন

Status:
- Open
